3. External Interface Requirements

3.1 User Interfaces

The opening page will have two buttons for two portals. One for the faculty members and the other for the students.

The faculty portal will use image verification, username, and password for logging in. Same applies for the students.

The faculty will be able to click on student-cohorts("Sections"), and will be allowed to click on individual students to provide them "Reading Assignments", set the time and set "Quizzes" and time for them as well.


They will receive detailed reports of their performances upon completion of the assignments on the "Reports" section.

This includes their aggregate results that is their average in a histogramic form based on their performance on "Quizzes".

More on that later. The faculty will be able to give "Reading Assignments" to all students instead of individually if they so choose.

The student portal will contain "Reading Assignments" for them and upon clicking they will be shown the assignments and time for them.

And they'll be able to take quizzes at their leisure and will be shown the time during each "Quiz."

3.2 Hardware Interfaces

The minimum requirements that have been discerned is that the product should work on the web via sign in through google, as well as be incorporated as a fully functioning mobile app to be used via android, iphone, watch etc.

3.3 Software Interfaces

The product shall be built using traditional web development tools, such as html, JavaScript, CSS and for database php as well as their frameworks and libraries such as laravel, bootstrap, react.js and others. Language for database manipulation such as MySQL will be used as information about faculty and students will need to be stored. Data from the database will be used to provide reports using internal algorithms for calcualations.

3.4 Communications Interfaces

The product will require internet connection to work properly. Users will need to sign in using their respective google accounts and will have the option to log out of the system. Faculty members will be able to register themselves as well as their students into the database at their convenience.

4. System Features

This section describes our product's most prominent features and characterizes their importance in the total completion of the product

4.1 Facial recognition system

4.1.1 Description and Priority

The system should allow for facial recognition of the users in order for them to log in to the system and must be utilized for the "Reading Assignments" and "Quizzes." It must also be able to check whether or not the students is looking forward at the screen as well, if possible. High priority with a rating of 8 from 1 to 9.

4.1.2 Stimulus/Response Sequences

During logging in, registration(for comparison during future log ins), reading assignments, quizzes, this feature will be utilized. It will receive input from the camera for its proper functioning. It will also check if someone's looking at the screen or not. Vital for the desired "Attention Score."

4.1.3 Functional Requirements

The software responsible for this feature is TBD. The program will not allow the user to log in or will kick them out in the case of an error. The functionality of this feature is of utmost importance.

4.2 PDF divider

4.2.1 Description and Priority

One of the requirements identified is that the product must have a feature which will take an uploaded pdf file and split it into subpages for the "Reading Assignments" section and must not allow the assignment to be downloaded in any way shape or form. Medium priority with a rating of 5.

4.2.2 Stimulus/Response Sequences

This software will be used when the faculty uploads their materials in pdf form for the assignments that will be assigned for the respective students.

4.2.3 Functional Requirements

The software for the feature is TBD. In the case of errors the feature ought not to allow for the pdfs to be uploaded.

4.3 Text Comparison System

4.3.1 Description and priority

The product must have a text comparison system which will allow for the internal algorithm itself to score quizzes(Whether or not the answers are right), but also allow for the program to identify the specific subpages from which the questions were made. The data will be used for the convenience of the faculty in the "Reports" section of their respective portals. AI may be used in the case of general questions instead of just ony MCQs. High priority with a rating of 8.

4.3.2 Stimulus/Response Sequences

After the student has finished their quizzes the algorithm will check their answers against the reading material. Will identify the parts of the subpages for the relevant questions. The data will provide feedback for the faculty.

4.3.3 Functional Requirements

The text recognition software is yet to be decided (TBD). The feature must avoid errors as much as possible for a thorough evaluation of the quizzes.

4.4 General Features

4.4.1 Description and Priority

The product must have a google log in, log out system, must support database integration for the "Student-cohorts" , Must allow internal calculations for the aggregate "Attention-scores", "Quiz" performances, in the form of histograms if possible. Highest priority of 9.

4.4.2 Stimulus/Response Sequences

Log in records, general data, student-cohort details, report details(Assignments & Quizzes), pdf file upload system etc. general features must be added for the product to work as a properly functional website and as a mobile app.

4.4.3 Functional Requirements

These features can be implemented using html, CSS, Javascript, php, MySQL as well as their frameworks and libraries like laravel, react.js etc.

4.5 Voice Input System

4.5.1 Description and Priority

A voice input system is to be used for the students to answer their quizzes using just their voices. Low priority with a rating of 2.

4.5.2 Stimulus/Response Sequences

It is to be used only if the faculty specifies that the answers will be taken through voice.

4.5.3 Functional Requirements

The voice input software for this feature is TBD.

4.6 Timer

4.6.1 Descripiton and Priority

Timer to be used to calculate the student's attention span for the desired "Attention-score" as well as for the implementation of quizzes under deadlines. High priority with a score of 7.

4.6.2 Stimulus/Response Sequences

It is to be used during the reading assignments and in the case of quizzes. After the timer runs out students will no longer have access to the quiz and assignments.

4.6.3 Functional Requirements

The software or the implementation method of the timer is TBD.
