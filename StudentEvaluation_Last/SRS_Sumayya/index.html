1. Introduction
1.1 Purpose

The primary purpose to develop this system to optimize and evaluate students by taking online exams and detecting their mistakes. Teachers can create a quiz platform for a particular student cohort where the students can participate in online examination in the form of MCQ, text answers and voice answer. The system will automatically evaluate all the answers and generate marks. Then the system will also show the percentage of wrong answered questions and from which part of the reading materials those answered belongs.

1.2 Document Convention

The font used in this document is Arial. The text is made bolt for the highlighting purpose so that it can be easily differentiated. In this document, every functionality is important and every requirement has its own priority.

1.3 Intended Audience and Reading Suggestions

The document is meant for the developers, testers, designers and users or anyone willing to work for the system. This document will serve as a reference document for every team who are taking part in making the project. They will coordinate every activity that take place in the software engineering process and will be guided by Dr. Nabeel Mohammed.

1.4 Product Scope

This Online Student Evaluation System enables users to have an exam taking facilities. It solves the typical issues of manually conducting examinations where both teachers and students have to attend. This system also showed that how much time students have spent behind the reading materials to get it. This kind of system is specially designed for educational institution to help in taking examination smoothly and judge the students level of studying properly.

1.5 References

Laravel
Bootstrap
Google API

2. Overall Description
2.1Product Perspective

Online Student Evaluation System is created to analyze the students over the perspective that how much time they have spent behind reading the study materials and how much they were able to understand it. Teachers can log into their accounts and create an exam portal for a student cohort where students can take part in exams after finishing their reading materials. The system is self-contained.

2.2 Product Functions

The system’s functions are super user-friendly so that it is easy for the users to operate. The more functions of the software are provided below:

  • All students can create an account and log in using google account.
  • Teacher who have created an exam portal will have editing abilities.
  • Students can sit for the exam and have to finish it within the time that has been assigned for the exam.
  • Teachers can see the marks, right answers and the histogram of the result over a student cohort.

2.3 User Classes and Characteristics

All students who are part of a particular cohort can log in after verifying their identity in video also. There are two different type of actors, they are teachers and students. Teachers can see the answers of questions, quiz marks, the improvement rates through histogram, upload reading materials. Students can see the reading materials but cannot download it and after finishing reading they will give exam that based on those reading materials.

2.4 Operating Environment

Platform supported: Web and Android
Web server: Any operating system with Laravel 4.2.17
Android App: Any system with a functional web browser
Database configuration: SQLite

2.5 Design and Implementation Constraints

We will have a web app, android app, face recognition app, voice message taking app for the system. Users will need to use these apps to get services from the system.

2.6 User Documentation

The product will come with a user manual which will explain how to use the system such as how to create a student-cohort, how to take exams, how to give answers of exam questions, how to sit in front of camera while video needs to be on.

2.7 Assumption and Dependencies

Users must have an active internet connection. Web users must have a working browser to use the system.

4.System Features
4.1 Log in Using Google

  • Use Case: Login using Google
  • Actor(s): Teachers, Students
  • Brief Description: The students and teachers can login to this system using their Google account and do the works those are assigned for them particularly.
  • Type: Primary

Typical course of events

Actor Action System Response
Users will choose the option between teacher and student. System redirect to the page of teacher or student.
User click the log in page if account already exist. Systems redirects to the google login page.
User put the sufficient data using google account. System will be responsive by verifying the credentials data through google API.

Alternative course of events

Section-1: If the user enters incorrect google credentials, the system will redirect to login page.

4.2 Create student-cohort

  • Use Case: Create student-cohort
  • Actor(s): Teacher
  • Brief Description: After login into the system, the teacher can create student-cohort by clicking the “create-student-cohort” button.
  • Type: Primary

Typical course of events

Actor Action System Response
Teachers clicks on “create-student-cohort” button. The system provides a form for creating student-cohort
Teachers fills up the necessary information and clicks submit button. The system will show a successful message and the cohort will be added to the dashboard.

Alternative course of events

Section-2: The system will show an error message if the form is not filled properly or if any information is missing it shows the message “the filled is required”.

4.3 Add Students to student-cohort

  • Use Case: Add students to a cohort
  • Actor(s): Teacher
  • Brief Description: The teacher can add students to the exam cohort so that they can have access to reading materials and give exams.
  • Type: Primary

Typical course of events

Actor Action System Response
Techer clicks on “add-student” button. System redirects to add student page
The teacher will enter the students email address and then click on “submit” button. The system will add the student to that cohort.

Alternative course of events

Section-3: There can be an error because the entered email might be invalid.

4.4 Upload Reading materials

  • Use Case: Upload reading materials as pdf
  • Actor(s): Teacher
  • Brief Description: The teacher will upload reading materials for students and later take exam over these.
  • Type: Primary

Typical course of events

Actor Action System Response
Teacher will click on a button “upload-materials-pdf” The system will allow teacher to upload only pdf materials.
Teacher will confirm the action System will show a successful message

Alternative course of events

Section-4: The button “upload-materials-pdf” will be shown only to teachers, not to students.

4.5 Reading Materials Divided into Segments

  • Use Case: Each pdf will be divided into suitable no of segments
  • Actor(s): The System
  • Brief Description: To identify which questions belongs to which part of the reading materials and give an overview to teacher about the students’ current position.
  • Type: Primary

Typical course of events

Actor Action System Response
The system will generate a percentage over the segments right answered questions The system will generate a histogram of a cohort over the percentage.

Alternative course of events

Section-5: The teacher will inform the system which question is made from which segment.

4.6 Create Exam Portal

  • Use Case: Creating an exam portal
  • Actor(s): Teacher
  • Brief Description: After login as a teacher, he can create an exam portal for a cohort by clicking on “create-exam-portal” button.
  • Type: Primary

Typical course of events

Actor Action System Response
Teacher clicks on “create-exam-portal” button. The system will provide a form to create it.
Teacher fills up the necessary information and clicks submit button. System will show a successful message and add an exam-portal for that student-cohort.

Alternative course of events

Section-6: The system will show an error message if the form is not filled properly or if any information is missing it shows the message “the filled is required”.

4.7 Add 3 types of Questions in Exam-portal

  • Use Case: Add MCQ, text answered questions, voice answered questions.
  • Actor(s): Teacher
  • Brief Description: Teacher will make some questions of these three types in a student-cohort and specify the correct answer and also time duration of the exam.
  • Type: Primary

Typical course of events

Actor Action System Response
Teacher will get some options to choose and click on “Add-MCQ”, “Add-text-Ques”, “Add-voice-Ques”. They system will let the teacher to create an form in mixed of 3 types patterns of Questions
Teacher will specify the answers. Students cannot see the answers and marks the got but teacher can.

Alternative course of events

Section-7: If teacher selects two options as correct answers of MCQ then an error message will pop up and the MCQ will not be formed.
Section-8: For voice answered question teacher will set some key words only to detect it as write answers same for text answered questions too.

4.8 Take participation in the exam

  • Use Case: Participating in exam
  • Actor(s): Students
  • Brief Description: The students of a student-portal can participate in exam after finishing the reading materials.
  • Type: Primary

Typical course of events

Actor Action System Response
This use case starts when a student clicks on “Start-Exam” The system will redirect the student to the exam window.
The students will click on one of the option of MCQ After clicking the submit button, system will save all the answers
The students will enter in text-Questions and answers the questions writing some text The system will save the answers after clicking on submit button.
The students will enter in voice-Qustions page and turn on mike to answer the questions. The system will detect the voice and save the answers.

Alternative course of events

Section-9: Answers cannot be edited after clicking the submit button.
Section-10: Students cannot turn of video while giving the exam otherwise the exam will be disqualified.

4.9 Face Recognition on Video

  • Use Case: Camera on
  • Actor(s): Students
  • Brief Description: Student will have to turn on the camera while logging or reading or giving exams.
  • Type: Primary

Typical course of events

Actor Action System Response
The use case starts when student clicks on “Log In”, “Log Out”, “Strat-Exam” button. System will ask to turn on camera.
To open reading materials System will ask to turn on the camera.

Alternative course of events

Section-11: If video turned off, it will show an error message to turn on the video.

4.10 Log Out

  • Use Case: Log out
  • Actor(s): Student, Teacher
  • Brief Description: The users can log out from the system by clicking on “Log Out” button.
  • Type: Primary

Typical course of events

Actor Action System Response
Student or Teacher clicks the “Log Out” button. The system will redirect to the login page.

5 Other Nonfunctional Requirements
5.1 Performance Requirements

  • Android app and web app will be using the same backend which ensures good performance and synchronization.
  • The system is well optimized.
  • The system is available 24 by 7

5.2 Safety Requirements

  • Errors will be minimized and an appropriate error message that guides the user from an error to fix the error will be provided.
  • Validation and authentication of user input is highly essential.

5.3 Security Requirements

  • The system ensures a high level of security and integrity of the data.
  • Only authorized personnel can have access to the private data and only the user with a valid google account is allowed to log in and view its user page.

5.4 Software Quality Attributes

  • The key software quality attributes are availability, reliability and usability.
  • Since the system is expected to be working 24/7, high availability and is important.
  • A quality and understandable user interface need to be developed.
  • The error messages have to be descriptive and can easily be understandable.

5.5 Business Rules

  • System should be available for all of its users.
  • All the users shall access the system using a valid google account.

Appendix A: Glossary

API:
It stands for Application Protocol Interface. An API is basically a software intermediate that allows two applications to interact with each other.

Framework:
A software framework is an abstraction in which software, providing generic functionality, can be selectively changed by additional user-written code, thus providing application-specific software.

Library:
A library is a collection of non-volatile resources used by computer programs, often for software development.

Android:
Android is a mobile operating system based on a modified version of the linux kernel and other open source software, designed primarily for touchscreen mobile devices.

Database:
A database is an organized collection of structured information, or data, typically stored electronically in a computer system.
